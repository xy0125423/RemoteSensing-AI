# STUDY_STRATEGY — 学习路线设计逻辑

> 本文档解释：为什么学习顺序是这样安排的？  
> 为什么不是先学 AI？为什么不是先学傅里叶？  
> 整个学习路线的设计逻辑是什么？

---

## 1. 核心设计理念：爬山，而不是坐电梯

```
大多数人的想法（错误）：
  学理论 → 学框架 → 学模型 → 做项目 → 完成
  问题：前 3 个月都在学理论，项目没动工，动力耗尽

本项目的设计（正确）：
  Day 1 → 就在 GEE 筛选影像 → 看到成果
  Week 2 → 就有了自己的 GeoTIFF
  Month 1 → 做出了第一张 NDVI 图
  Month 3 → 有了完整的时序分析
  Month 6 → 一键生成 PDF

  每个阶段都有"看得见的成果物"
```

**心理学原理**：正向反馈循环（Positive Feedback Loop）。  
每完成一个 Task，你看到实实在在的输出（一张图、一个 CSV、一个 PDF），这会驱动你继续下一个 Task。

## 2. 为什么 Phase 1 是 GeoTIFF，不是 NDVI？

### 2.1 从"数据"开始，而不是从"分析"开始

```
数据 ← 你在这里开始
 ↓
处理
 ↓
分析
 ↓
结论
```

遥感开发的真实流程是：
1. 先拿到数据（没有数据什么都算不了）
2. 再理解数据格式（GeoTIFF 是什么，里面有什么）
3. 才能做分析（NDVI、时序、FFT）

如果反过来：
- 先学 NDVI 理论 → 没有数据练手 → 概念抽象 → 容易放弃
- 先学 FFT 理论 → 没有实际的信号 → 纯数学 → 枯燥

### 2.2 遵循遥感数据处理的标准流程

```
空间筛选 → 时间筛选 → 质量筛选 → 导出 → 本地读取 → 分析
   ↑                                                ↑
 Phase 0 (已做)                              Phase 1 (正在做)
```

这个流程不是我自己发明的，而是遥感领域的事实标准。  
你学会了这个流程，以后处理 Landsat、MODIS、Sentinel-1（SAR）都是同样的思维。

---

## 3. 为什么 NDVI (Phase 2) 在时间序列 (Phase 3) 之前？

### 3.1 NDVI 是时间序列的"原材料"

```
Phase 2: NDVI 计算
  ├── 输入：28 景 GeoTIFF
  └── 输出：28 个 NDVI 值

Phase 3: 时间序列分析
  ├── 输入：28 个 (日期, NDVI) 对
  └── 输出：平滑曲线 + 物候指标
```

你不可能在没有 NDVI 值的情况下做时间序列分析。  
这是最直接的依赖关系。

### 3.2 NDVI 更简单，更容易获得正反馈

| 对比维度 | NDVI (Phase 2) | 时间序列 (Phase 3) |
|----------|----------------|---------------------|
| 公式复杂度 | (A-B)/(A+B) | 插值 + 平滑 + 分解 |
| 输出类型 | 一张图 | 一条曲线 |
| 是否正确 | 目视即可判断 | 需要统计验证 |
| 成就感 | "我会算植被指数了！" | "曲线好像不太对..." |

先做简单的，获得信心，再做复杂的。

## 4. 为什么 FFT (Phase 4) 放在时间序列 (Phase 3) 之后？

### 4.1 FFT 的前置条件：均匀采样信号

```python
# FFT 要求
ndvi_signal = [0.35, 0.37, 0.40, ...]  # 均匀时间间隔！
dt = 1  # 步长为 1 天

# 但原始 NDVI 序列是：
ndvi_raw = {
    '2025-03-15': 0.35,
    '2025-03-25': 0.42,  # 间隔 10 天
    '2025-04-01': 0.55,  # 间隔 7 天
    '2025-04-16': 0.63,  # 间隔 15 天（云太多，缺失）
    ...
}
```

如果 Phase 3 没有做插值 → 没有均匀序列 → FFT 结果不可靠。  
这是硬依赖，不能跳过。

### 4.2 FFT 是"分析的深化"，不是"分析的起点"

```
Phase 2 (NDVI):   看到"这里植被多，那里植被少"
Phase 3 (时序):    看到"春天变绿，夏天最绿，秋天变黄"
Phase 4 (FFT):    看到"为什么变？周期是多久？多季作物吗？"
```

每深入一层，你对数据的理解就深一层。  
如果跳过 Phase 2 和 Phase 3 直接做 FFT，你会得到一堆频率，但完全不知道它们代表什么。

---

## 5. 为什么可视化+PDF (Phase 5) 放在最后？

### 5.1 先有"内容"，再考虑"包装"

```
错误顺序：先设计 PDF 模板 → 再想往里面填什么
正确顺序：先做完所有分析 → 再设计 PDF 怎么展示
```

如果你在 Phase 1 就设计 PDF 模板，你会：
- 预设一些可能根本不需要的章节
- 花大量时间调排版，而分析结果还没出来
- 后期分析结果变了，PDF 模板可能要大改

### 5.2 但可视化不能等到最后

虽然"美化"放在 Phase 5，但"基础可视化"贯穿所有 Phase：

| Phase | 可视化内容 | 目的 |
|-------|-----------|------|
| Phase 1 | 真彩色影像 | 验证数据正确 |
| Phase 2 | NDVI 伪彩色图 | 验证 NDVI 计算正确 |
| Phase 3 | 时序折线图 | 验证序列合理性 |
| Phase 4 | FFT 频谱图 | 验证频谱正确 |
| Phase 5 | 美化所有图 | 达到报告质量 |

"先画出来看看对不对" → 这是调试。  
"画得好看放进报告" → 这是美化。  
两个阶段，不能混为一谈。

## 6. 为什么不是先学 AI / 深度学习？

### 6.1 硬件限制是硬约束

```
你的电脑：Lenovo 小新16，无 NVIDIA GPU
深度学习：需要 GPU，训练一个 ResNet 可能需要几小时到几天 (GPU)
          在 CPU 上训练可能需要几周，且笔记本散热撑不住
```

### 6.2 数据量不匹配

```
深度学习：需要几千到几百万标注样本
本项目：28 景 Sentinel-2 影像
→ 用 DL 是杀鸡用牛刀，而且鸡都找不到
```

### 6.3 本科阶段的学习目标不是"会用框架"

```
会用 PyTorch = 调包侠（任何人花一周都能学会）
理解遥感原理 + 信号处理 + 工程能力 = 独特竞争力（需要长期积累）
```

复试导师问的不是"你会不会调 torch.nn.Conv2d"，而是：
- "你为什么用 NDVI 而不是 EVI？"
- "NDVI 在浓密植被区为什么会饱和？"
- "你从频谱图中读出了哪些信息？这和采样定理有什么关系？"

这些问题的答案，来自 Phase 2 和 Phase 4 的扎实实践，而不是跑一个 CNN。

### 6.4 传统方法是深度学习的"地基"

```
你在本项目学到的               未来可以延伸到
─────────────────────         ─────────────────
NDVI 计算                     深度学习的地面真值标签
时间序列分析                   LSTM 时序预测
FFT 频谱分析                   深度学习模型的特征工程
GEE 数据流水线                 DL 训练数据的自动获取
模块化工程代码                 ML Pipeline (Kubeflow 等)
```

先打好地基，以后盖高楼。没有地基的高楼 = 摇摇欲坠。

## 7. 为什么 GEE 数据获取放在最前面？

### 7.1 数据是遥感开发的"入场券"

```
没有数据 → 什么都做不了
有数据 → 哪怕只算一个均值也有意义
```

Phase 0 花了 2 天把 GEE 数据流跑通，这不是浪费时间——  
这 2 天保证了后续 5 个月都有"米"下锅。

### 7.2 GEE 降低了数据获取的门槛

```
传统方式：
  1. 去 ESA SciHub 搜索数据
  2. 等待审核
  3. 下载（单景 1GB+，容易断连）
  4. 手动解压
  5. 手动大气校正（需要 Sen2Cor）
  → 一个本科生可能卡在第二步一周

GEE 方式：
  1. 写 10 行代码
  2. 云端自动处理
  → 10 分钟搞定
```

选择 GEE 不是为了偷懒，而是把你有限的时间用在"分析"上，而不是"下载"上。

## 8. 整个学习路线的"暗线"：与考研同步

学习路线不只是项目的逻辑，还有一条暗线——考研 892 信号与线性系统：

```
Phase 0-1 (7月)    项目打基础          考研：基础复习
Phase 2   (8月)    NDVI + 批量处理      考研：第一轮
Phase 3   (9月)    时间序列 + 采样      考研：采样定理复习 ← 项目触发
Phase 4   (10月)   FFT + 频谱分析       考研：FFT 重点突破 ← 高度重合！
Phase 5   (11月)   可视化 + PDF         考研：冲刺
Phase 6   (12月)   集成 + 打磨          考研：初试
```

注意 10 月的设计：
- 项目的 Phase 4 是 FFT
- 考研 892 的 FFT（第 3~5 章）正好也是 10 月重点复习内容
- 这意味着你**做项目 = 复习考研**，不是零和博弈

## 9. 为什么不建议跳过任何 Phase？

### 9.1 每个 Phase 都是下一个的"地基"

```
如果跳过 Phase 1（GeoTIFF）→ Phase 2 不知道怎么读取数据
如果跳过 Phase 2（NDVI）→ Phase 3 没有数据来源
如果跳过 Phase 3（时序）→ Phase 4 没有均匀采样信号
如果跳过 Phase 4（FFT）→ 项目失去与考研的连接点
如果跳过 Phase 5（PDF）→ 项目没有交付物，复试展示不了
如果跳过 Phase 6（集成）→ 一堆孤立的脚本，不是"系统"
```

### 9.2 但每个 Phase 内部的 Task 可以调整优先级

```
必须做（Core Task）：Task 01, 03, 05, 09, 12, 15, 17, 21, 22, 28, 32, 36
可以做（Nice to Have）：Task 11, 13, 20, 25
考研紧张时可跳过：Task 20, 25, 38（测试）, 40（3份报告缩为1份）
```

## 10. 三条学习原则

### 10.1 先求完整，再求完美

```
Task 09 的 NDVI 计算：先用 numpy 最朴素的实现 → 能跑 → 再优化
不要：先研究 3 天"最优 NDVI 算法" → 一行代码没写 → 放弃
```

### 10.2 每个概念都要经过"手写验证"

```
❌ 调用 rasterio 的现成函数 → "NDVI 算出来了"
✅ 手动选一个像素，用计算器算 (NIR-Red)/(NIR+Red) → 和代码输出对比 → 一致！
```

手写验证一次，比你读十篇教程都管用。

### 10.3 笔记驱动学习

```
每完成一个 Task → 在 NOTES.md 写 5 句话：
  1. 做了什么
  2. 遇到什么问题
  3. 怎么解决的
  4. 学到了什么
  5. 下次注意什么
```

这些笔记是你复试时最宝贵的"故事素材"。

---

## 11. 总结：一句话的设计哲学

```
不是"先学会再做"，而是"边做边学"。
不是"一步到位"，而是"迭代优化"。
不是"追求技术先进性"，而是"追求理解的深度"。
```

六个月后，你最大的收获不是一份 PDF 报告，而是：
- 知道遥感数据从哪来
- 知道怎么把它变成有用的信息
- 知道怎么用信号处理的方法分析它
- 知道怎么把分析结果讲清楚
- 知道怎么管好一个软件项目

这才是考研复试导师真正想看到的"科研潜力"。
